게임의 초기 상태 설정

Author: 9um4 Date: 8월 21, 2025  1:04:29

배경

사용자가 Application을 실행하는 동안 전역적으로 관리해야 하는 Context들이 있습니다. 예로 들어 Player의 Main Party에 포함되어 있는 Character들에 대한 정보나 현재 Application의 State가 대표적인 예시가 될 수 있을 것 같습니다.

Unity 내에서는 일반적으로 GameManager라는 Class를 구현하고, 해당 Class에 Context를 Attribute의 형태로 담거나, 혹은 GameManger가 관리하는 하위 Class들에 해당 Context들을 나누어 관리함으로서 해결합니다. 하지만 GameManager를 일반적인 Component를 구현하는 것 처럼 구현할 경우 몇 가지 문제를 유발할 수 있습니다.

1.  `GameManager`로부터 유래한 Instance가 2개 이상일 경우 각각의 Instance들은 각자의 Context를 가질 수 있다는 문제가 발생합니다. 
2. 설계할 때 `GameManager`는 Unity의 MonoBehaviour를 상속하기 때문에 `GameManager`가 생성되고 유지할 수 있는 환경을 만들어 주는 것이 필요합니다.

이런 문제들을 해결하기 위해 Application의 Life Cycle 동안 단 하나만 존재하고 파괴되지 않는 특별한 환경을 GameManager, 필요한 경우 다른 Manager Instance들에게 제공하는 구조를 설계하려고 합니다.

구현

Singleton Pattern을 통한 유일성 보장

앞서 언급한 바와 같이 GameManager로부터 유래한 Instance가 2개 이상일 경우 각각의 Instance들은 각자의 Context를 가질 수 있습니다.

  • 이를 방지하기 위해 Singleton Pattern을 활용해 GameManager Class로부터 유래한 Instance를 하나만 만들고, 두 번째 이후의 생성 시도는 생성하지 못하게 끔 조건을 나눔으로서 구현합니다.
  • Class 외부에서 static이 아닌 public Attribute를 접근하기 위해서는 static Attribute인 Instance를 통해 접근합니다.

GameManager를 Static Class로 선언하지 않은 이유

GameManager의 역할을 생각하면 GameManager를 Static Class로 선언하더라도 큰 문제가 없어 보입니다. Static Class일 경우 굳이 GameManager 내부의 Static Field인 GameManager#Instance를 통해 Attribute를 접근할 필요가 없어 오히려 더 나은 것 처럼 느껴집니다. 하지만 GameManager가 Static Class일 경우 Static Class가 아닌 MonoBehaviour를 상속할 수 없기 때문에 하술할 문제가 발생합니다.

  1. Unity Lifecycle을 활용할 수 없습니다.
    • Static Class는 Unity의 Life Cycle Event Method인 Awake(), Start(), Update()를 사용할 수 없습니다.
    • 이럴 경우 GameManager의 내부에서 Application의 Life Cycle Event Method를 구현해야 하는데 이를 구현하는 과정에서 오류가 발생할 가능성을 무시할 수 없습니다.
  2. Inspector 연동이 불가능합니다.
    • Static Class는 Unity의 MonoBehaviour Class를 상속할 수 없기 때문에 GameObject에 Component를 붙일 수 없습니다. 따라서 Static Class의 Attribute의 값을 Inspector 창에서 수정할 수 없습니다.

따라서 GameManager를 Static Class로 구현하는 것 보다 Singleton Pattern을 이용해 한 번 GameManager가 Initialize된 후의 Context를 잃어버리지 않도록 구현하였습니다.

Application의 초기 Context 설정

GameManager를 Singleton Pattern을 이용해 구현하였지만 이 GameManager의 Context는 게임이 시작되지마자 자동적으로 Initailize가 일어나지 않습니다. Static Class가 아니기 때문에 최초의 Instance를 생성해주는 작업이 수반되어야 합니다.

따라서 최초의 GameManager로부터 유래한 Instance의 생성을 안정적으로 보장하기 위해 최초 환경을 설정하는 역할만 담당하는 @init Scene을 선언합니다. 그 후 해당 Scene에 핵심 System Manager를 Component로 가지는 GameObject를 배치하고 초기 설정이 되어 있지 않을 경우 해당 Scene을 Load함으로서 Application의 관점에서 초기 Context를 마련할 수 있습니다.

Load된 후 @init Scene 내에 존재하는 Component들을 DontDestroyOnLoad Scene으로 옮겨 Application이 실행되는 동안 파괴되지 않도록 합니다.

이 Logic은 GameManager과 별개인 GameInitializer라는 Application의 초기 Context를 설정해주는 Class에 구현합니다.

Property를 이용한 Method의 실행 타이밍의 보장

RuntimeInitializeOnLoadMethod(RuntimeInitializeLoadType.AfterSceneLoad) Property를 이용해 Scene Load가 완료된 직후 자동으로 실행하도록 보장하고자 합니다.

더 생각해 보아야 할 부분

GameInitializer는 게임이 어떤 상황에서 시작되더라도 안정적으로 초기 Context를 설정해주기에 상대적으로 더 안정적인 Application의 동작을 보장할 수 있습니다. 하지만 개발자가 특정 기능 테스트를 위해 BattleScene이나 LobbyScene에서 바로 작업을 시작하는 경우, 이 로직은 현재 작업 중인 씬을 강제로 @Init 씬으로 전환시거나 혹은 Manager Class에 구현되어 있는 Initialize 로직이 실행되면서 개발 과정에서의 테스트를 어렵게 만듭니다.

  • 해당 Manager에서 Unity Editor 상에서 Game을 실행했을 때 Initialize Logic을 실행 시킬지 말지 하는 Flag Variable을 선언해 흐름을 제어합니다.
  • 혹은 각 Manager의 초기화 로직의 실행 여부를 제어하는 Flag Variable을 관리하는 것을 하나의 ScriptableObject의 책임으로 하고 각 Manager에서 해당 Flag를 읽어 와 수행하는 방식으로 구현할 수도 있습니다.
